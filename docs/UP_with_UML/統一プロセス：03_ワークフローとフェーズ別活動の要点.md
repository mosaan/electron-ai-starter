本文書では、統一プロセスにおける主要な基本ワークフローごとの「作業」（アクティビティ）と、各フェーズ（方向づけ、推敲、作成、移行）においてそれらの作業がどのように重点的に実施されるかをまとめます。

統一プロセスは、要求、分析、設計、実装、テストの5つの基本ワークフローから構成され、これらは各反復（ミニプロジェクト）の中で繰り返されます。

### 統一プロセスの基本ワークフローと主要な作業

#### 1. 要求ワークフロー (Requirements Workflow)

要求ワークフローの目的は、適切なシステムに焦点を合わせるために、システム要求を正しく記述し、顧客と開発者の間で取り決めを行うことです。

| 作業 (Activity) | 概要 |
| :--- | :--- |
| **アクターとユースケースの洗い出し** | システムの境界を定め、システムと相互作用する人やモノ（アクター）と、システムに求められる機能（ユースケース）を明確にする。 |
| **ユースケースの優先度づけ** | 初期反復で開発すべきユースケース（アーキテクチャ上重要なものなど）と、後続反復で開発すべきものを判断するための材料を用意する。 |
| **ユースケースの詳細化** | ユースケースの開始・終了方法やアクターとの相互作用を含むイベントフローを詳しく記述する。 |
| **ユーザーインターフェイスのプロトタイプ作成** | アクターがユースケースを効果的に実行できるようにする論理的・物理的なユーザーインターフェイスを設計し、プロトタイプまたはスケッチを作成する。 |
| **ユースケースモデルの構築** | ユースケースの汎化や拡張などのメカニズムを採用し、構造化された、わかりやすいモデルにする。 |

| フェーズごとの重点 | 補足 |
| :--- | :--- |
| **方向づけフェーズ** | このフェーズの作業の**ほとんどが要求ワークフローで完成**します。特に、**システムの責任範囲**を設定できるかどうかに影響する**ユースケースの一部**（全体の10%程度）と、**致命的リスクの理解に関わるユースケース**の洗い出しと詳細化に注力します。 |
| **推敲フェーズ** | **要求ワークフローの作業量は依然として多い**ですが、方向づけフェーズで開始した作業を拡張します。**ユースケースの約80%を把握**し、アーキテクチャにとって重要なユースケースの詳細化に労力を集中します。 |
| **作成フェーズ** | **作業負荷はかなり低くなります**。残りのユースケースとシナリオの詳細化を完了し（最終的に100%を目指す）、**ユーザーインターフェイスのプロトタイプ作成**が重要になる場合があります。 |
| **移行フェーズ** | ほとんど作業はありません。

#### 2. 分析ワークフロー (Analysis Workflow)

分析ワークフローの目的は、要求を改良し、システムの構造を理解する手がかりとなる概念的なオブジェクトモデル（分析モデル）として構造化することです。

| 作業 (Activity) | 概要 |
| :--- | :--- |
| **アーキテクチャ分析** | 分析モデルとアーキテクチャの概要を捉えるために、主な分析パッケージ、明白なエンティティクラス、共通する要求を洗い出す [155, 8.6.1節]。 |
| **ユースケースの分析** | ユースケースのイベントフローを実行するのに必要なオブジェクトを持つ分析クラスを洗い出し、ユースケースの振る舞いを相互作用する分析オブジェクトに分散する [158, 8.6.2節]。 |
| **クラスの分析** | ユースケースの実現での役割に基づいて分析クラスの責務、属性、関係、特殊要件を洗い出し、維持する。 |
| **パッケージの分析** | サービスやクラスをサービスパッケージとしてグループ化し、パッケージの改良と維持を行う。 |

| フェーズごとの重点 | 補足 |
| :--- | :--- |
| **方向づけフェーズ** | 作業は最小限にとどまります。**アーキテクチャ候補の説明**のために、最初の荒削りの分析モデル（分析モデルのアーキテクチャビューへの最初の一歩）を作成します。 |
| **推敲フェーズ** | **推敲フェーズ初期の反復の中心的な作業**です。**堅実で安定したアーキテクチャを構築する**手助けとなります。**実行可能なアーキテクチュラルベースライン**に到達するために、アーキテクチャにとって重要なユースケースを分析します。詳細に記述したユースケース群全体の約50%を調べる必要があるでしょう。 |
| **作成フェーズ** | 分析モデルを完成させます。推敲フェーズで完了したアーキテクチャ分析の更新を除けば、アーキテクチャに影響する変更によって必要となる更新を除いて、**作業量は低くなります**。 |
| **移行フェーズ** | ほとんど作業はありません。

#### 3. 設計ワークフロー (Design Workflow)

設計ワークフローの目的は、実装に必要な入力を作成し、システムの枠組み（アーキテクチャを含む）を形づくり、機能外要求や制約を支えることです。

| 作業 (Activity) | 概要 |
| :--- | :--- |
| **アーキテクチャ設計** | 設計モデル、配置モデル、およびアーキテクチャの概要を作成するために、ノード/ネットワーク構成、サブシステム/インターフェイス、アーキテクチャ上重要な設計クラス、汎用設計メカニズムを洗い出す。 |
| **ユースケースの設計** | ユースケースのイベントフローを実行するのに必要な設計クラスやサブシステムを洗い出し、振る舞いを相互作用する設計オブジェクトやサブシステムに分散する。 |
| **クラスの設計** | 設計クラスの操作、属性、関係、実装要求、メソッド（アルゴリズム）の仕様を定義する。 |
| **サブシステムの設計** | サブシステムが提供するインターフェイスと、その内部でインターフェイスを提供する設計クラスまたはサブシステムを定義する。 |

| フェーズごとの重点 | 補足 |
| :--- | :--- |
| **方向づけフェーズ** | **アーキテクチャ候補を予備的なアーキテクチャ説明書に組み入れる**ために、設計モデルを略述するのが主な目標です。実装が必要な場合（実証プロトタイプ）、設計範囲は最小限にとどまります。 |
| **推敲フェーズ** | **推敲フェーズ後半の反復の中心的な作業**です。**アーキテクチュラルベースライン**を確立するため、**アーキテクチャ設計**（階層化アーキテクチャ、設計メカニズム、ノード構成の洗い出し）に力を注ぎます。アーキテクチャ上重要なユースケースを設計します。 |
| **作成フェーズ** | **重要な焦点**となります。アーキテクチュラルベースラインで確立された**骨格に肉づけする作業**が中心となり、残りのユースケースの"ユースケースの実現"を完成します。アーキテクトはアーキテクチャの整合性を維持し、必要に応じて更新します。 |
| **移行フェーズ** | 通常、設計作業は**後退**します。

#### 4. 実装ワークフロー (Implementation Workflow)

実装ワークフローの目的は、設計結果に基づき、ソースコード、実行可能ファイルなどのコンポーネントとしてシステムを実装し、単体テストと統合を行うことです。

| 作業 (Activity) | 概要 |
| :--- | :--- |
| **アーキテクチャ実装** | 実装モデルとそのアーキテクチャの概要を作成するために、アーキテクチャにとって重要なコンポーネントを洗い出し、ノードにマッピングする。 |
| **システムの統合** | 反復で必要なビルドの計画（統合化ビルド計画）を立て、各ビルドを統合し、統合テストに向けて準備する。 |
| **サブシステムの実装** | 設計サブシステムを対応する実装サブシステムとして実装し、各ビルドでその役割を果たすことを確認する。 |
| **クラスの実装** | 設計クラスをコンポーネント（ソースコード/ライブラリ/実行可能など）に実装し、操作をメソッドとして実装する。 |
| **単体テストの実行** | 各コンポーネントが単体として動作することを検証する（単体テストはコンポーネントエンジニアの責務）。 |

| フェーズごとの重点 | 補足 |
| :--- | :--- |
| **方向づけフェーズ** | 通常、実装ワークフローを続ける必要はありませんが、**概念立証プロトタイプ（使い捨てプロトタイプ）**が必要な場合に小規模に実行されます。 |
| **推敲フェーズ** | **実行可能なアーキテクチュラルベースラインを作成する**ために必要となります。**アーキテクチャ実装**、**システムの統合**が中心で、アーキテクチャにとって重要なコンポーネントが実装およびテストされます。 |
| **作成フェーズ** | **最もリソースを消費する作業**であり、**重要な焦点**です。**システムの構築**に集中し、全コンポーネントの実装と単体テストの実行を行います。**システムの統合**（ビルド単位でのインクリメンタルな統合）が鍵となります。 |
| **移行フェーズ** | 初期運用で発見された**欠陥の修復**（図10.2の移行の突出部）が中心となり、実装モデルは維持されます。

#### 5. テストワークフロー (Test Workflow)

テストワークフローの目的は、システムがユーザーにとって正しいユースケースを実現しているかを検証することです。

| 作業 (Activity) | 概要 |
| :--- | :--- |
| **テストの立案** | テスト方針、リソース、スケジュールを記述したテスト計画書を作成する [231, 232, 11.5.1節]。 |
| **テストの設計** | 各ビルドのテストケースを洗い出し、テストプロシージャ（実行方法）を編成する。 |
| **テストコンポーネントの作成** | 自動化されたテストプロシージャを含むテストコンポーネントを作成する [235, 11.5.3節]。 |
| **統合テストの実行** | 各ビルドに必要な統合テストを実行し、テスト結果を捉える（コンポーネントが正しく相互作用しているかを検証）。 |
| **システムテストの実行** | 統合されたシステム全体がユースケース通りに機能するかを検証し、欠陥を捉える。 |
| **テストの評価** | テスト作業の結果（カバレージ、欠陥のステータスなど）を評価し、次の反復にフィードバックする。 |

| フェーズごとの重点 | 補足 |
| :--- | :--- |
| **方向づけフェーズ** | **仮のテスト計画**を作成し、システムの一般的な性質に精通します。実証プロトタイプに対する小規模なテストが必要な場合はあります。 |
| **推敲フェーズ** | **実行可能なアーキテクチュラルベースラインのテスト**に集中します。**テスト環境の構築**、およびアーキテクチャ上重要なユースケースを実装するコンポーネントのテストに集中します。**統合テストの実行**と**システムテストの実行**がベースライン検証のために行われます。 |
| **作成フェーズ** | **このフェーズの重要な作業**です。単体テスト、統合テスト、システムテストが体系的に実行されます。テストケースが回帰テストとして使われるため、テストの計画と設計の成果が実ります。 |
| **移行フェーズ** | **ベータテストや受け入れテストで見つかる欠陥の修復と回帰テスト**に集中します。**テストの実行と評価**が作業の大半を占めます。

***

**補足：ワークフローの力点の移行**

統一プロセスでは、開発が進むにつれて、作業の重点が移り変わります。

1.  **方向づけフェーズ:** 要求の把握と予備的な分析/設計に労力のほとんどが向けられます。
2.  **推敲フェーズ:** 要求の把握の作業量は依然として多いものの、分析と設計のワークフローに集中し、アーキテクチャを確立します。
3.  **作成フェーズ:** 重点が**詳細設計、実装、テスト**に移り、この3つのワークフローが作業の大半を占めます。
4.  **移行フェーズ:** 欠陥の修復を保証するための実装と、**回帰テスト**に力点が置かれます。
