本文書では、統一プロセスにおける主要な基本ワークフロー（要求、分析、設計、実装、テスト）ごとに、そこで作成される主な**成果物（Artifact）**と、その成果物に対して**責任（責務）を負うワーカー（Worker）**を以下の表にまとめます。

統一プロセスにおいて、**ワーカー**とは開発要員に割り当てられ、彼らが受け入れる立場を表す用語であり、**成果物**とはシステムを開発する中でワーカーによって作成、変更、または使用されるあらゆる種類の情報に対する一般用語です。各ワーカーは一連の成果物に対して責任を持ちます。

### 各基本ワークフローにおける成果物と責任ワーカー

| 基本ワークフロー | 成果物（Artifact）| 責任を負うワーカー（Worker）| 備考/詳細 |
| :--- | :--- | :--- | :--- |
| **要求** (Requirements) | **特色リスト** (Feature List) | - | 要求候補のリスト。作業の立案に使われます。 |
| | **ビジネスモデル**または**ドメインモデル** (Business Model/Domain Model) | **ビジネスモデル作成者** (Business Modeler) | システムの外界を表現し、ビジネスユースケースやビジネスエンティティなどを含む。 |
| | **ユースケースモデル** (Use-Case Model) | **システム分析者** (System Analyst) | アクターとユースケースおよびそれらの関係を含んだシステムのモデル。要求全体に対して責務を負います。 |
| | **アクター** (Actor) | **システム分析者** (System Analyst) | システムと連携するシステム外部のものを表します。 |
| | **ユースケース** (Use Case) | **ユースケース定義者** (Use-Case Specifier) | アクターに価値をもたらすシステムの機能の一部を記述。個々のユースケースの詳細記述に対して責務を負います。 |
| | **用語集** (Glossary) | **システム分析者** (System Analyst) | 重要な概念や共通語彙を定義し、ユースケース記述や分析・設計で用語の一貫性を保つ。 |
| | **ユーザーインターフェイスのプロトタイプ** | **ユーザーインターフェイスデザイナー** (User-Interface Designer) | ユーザーインターフェイスの外観をデザインします。 |
| | **補足要求の仕様書** (Supplementary Specification) | **システム分析者** (System Analyst) | 特定のユースケースに特有ではない、一般的な要求（機能外要求など）を記述します [153, 6.7節参照]。 |
| | **アーキテクチャ説明書 (ユースケースモデルのビュー)** | **アーキテクト** (Architect) | ユースケースモデル中のアーキテクチャ上重要な要素を説明します。 |
| **分析** (Analysis) | **分析モデル** (Analysis Model) | **アーキテクト** (Architect) | 概念的なオブジェクトモデルであり、要求を改良するときに役立ちます。モデル全体の整合性を保証します。 |
| | **分析クラス**（Control, Entity, Boundary） | **コンポーネントエンジニア** (Component Engineer) | 1つまたは複数の分析クラスの責務、属性、関係、特殊要件の定義と維持に責務を負います。 |
| | **分析パッケージ / サービスパッケージ** | **コンポーネントエンジニア** (Component Engineer) | 1つまたは複数の分析パッケージの整合性の維持に責務を負います。 |
| | **ユースケースの実現－分析** (Use-Case Realization - Analysis) | **ユースケースエンジニア** (Use-Case Engineer) | 特定のユースケースを実現する方法を分析クラスのコラボレーションとして記述。その整合性に責務を負います。 |
| | **アーキテクチャ説明書 (分析モデルのビュー)** | **アーキテクト** (Architect) | 分析モデルのアーキテクチャにとって重要な部分を説明します。 |
| **設計** (Design) | **設計モデル** (Design Model) | **アーキテクト** (Architect) | 実装の青写真として機能するオブジェクトモデル。モデル全体の整合性を保証します。 |
| | **設計クラス** (Design Class) | **コンポーネントエンジニア** (Component Engineer) | 1つまたは複数の設計クラスの操作、メソッド、属性、関係、実装要求の定義と維持に責務を負います。 |
| | **設計サブシステム / サービスサブシステム** | **コンポーネントエンジニア** (Component Engineer) | 1つまたは複数のサブシステムの整合性を維持します。 |
| | **インターフェイス** (Interface) | **コンポーネントエンジニア** (Component Engineer) | サブシステムが提供するインターフェイスの仕様に対して責務を負います。 |
| | **ユースケースの実現－設計** (Use-Case Realization - Design) | **ユースケースエンジニア** (Use-Case Engineer) | 特定のユースケースを実現する方法を設計クラスのコラボレーションとして記述。その整合性に責務を負います。 |
| | **配置モデル** (Deployment Model) | **アーキテクト** (Architect) | システムを分散するネットワーク構成とノードを記述します。 |
| | **アーキテクチャ説明書 (設計モデル/配置モデルのビュー)** | **アーキテクト** (Architect) | 設計モデルおよび配置モデルのアーキテクチャ上重要な成果物を説明します。 |
| **実装** (Implementation) | **実装モデル** (Implementation Model) | **アーキテクト** (Architect) | コンポーネントに基づいて、設計要素をどのように実装するかを記述。 |
| | **コンポーネント** (Component) | **コンポーネントエンジニア** (Component Engineer) | 設計モデルの要素を物理的にパッケージ化する成果物。標準ステレオタイプ（executable/file/library/table/documentなど）を用いて性質を明示します。 |
| | **実装サブシステム** (Implementation Subsystem) | **コンポーネントエンジニア** (Component Engineer) | 1つまたは複数の実装サブシステムの整合性の維持に責務を負います。 |
| | **インターフェイス** (Interface) | **コンポーネントエンジニア** (Component Engineer) | 実装サブシステムやコンポーネントが提供・実現する操作を定義します。 |
| | **統合化ビルド計画** (Integration Build Plan) | **システム統合担当者** (System Integrator) | 各反復で作成する必要がある一連のビルドの計画と、各部分を実装したときのビルドの統合が含まれます。 |
| | **アーキテクチャ説明書 (実装モデルのビュー)** | **アーキテクト** (Architect) | 実装モデル中のアーキテクチャを理解する上で重要な成果物に対する説明。 |
| **テスト** (Test) | **テスト計画書** (Test Plan) | **テスト設計者** (Test Designer) | テストの実施方針、リソース、スケジュールを記述します。 |
| | **テストモデル** (Test Model) | **テスト設計者** (Test Designer) | テストの設計、テストの実施方針、リソース、スケジュールを定義します。 |
| | **テストケース** (Test Case) | **テスト設計者** (Test Designer) | 何をテストするかを指定し、検証可能な要求を検証するために考案されます。 |
| | **テストプロシージャ** (Test Procedure) | **テスト設計者** (Test Designer) | テストを実行する方法を指定します。 |
| | **テストコンポーネント** (Test Component) | **コンポーネントエンジニア** (Component Engineer) | 自動化された一部のテストプロシージャを含みます。 |
| | **欠陥** (Defect) | **統合テスト担当者** (Integrator Tester) / **システムテスト担当者** (System Tester) | テストの結果報告され、開発者が追跡し解決すべきシステムの問題の症状です。 |
| | **テスト評価書** (Test Evaluation Summary) | **テスト設計者** (Test Designer) | テスト作業の結果を評価したものです。 |

### 補足：ワークフローとワーカーの関わり

複数のワークフローを横断して関わるワーカーが多く存在します。例えば、**コンポーネントエンジニア**は分析、設計、実装の3つのワークフローに参加することがあります。また、**アーキテクト**は要求、分析、設計、実装の全てのワークフローでアーキテクチャ関連の成果物に対して責務を持ちます。

これは、統一プロセスが各反復において基本ワークフロー（要求、分析、設計、実装、テスト）をほぼ完全に実行し、ワーカーと成果物が参加しているコラボレーション（ワークフロー）として記述されるためです。
