「統一プロセス」（統一ソフトウェア開発プロセス）における**アーキテクチャ説明書（Architecture Description）**に求められる内容と、特に方向づけフェーズで焦点となる情報、および他のフェーズでの概略について包括的に説明します。

アーキテクチャ説明書は、システムの最も重要な静的な側面と動的な側面を具体的に表現したものであり、開発を統制する上で不可欠なシステム全体の明確な構図を捉えることができます。これは、開発者、管理者、顧客、その他の利害関係者に、何をなすべきかを示す詳しい内容を提示し、理解してもらうために最も必要とされる成果物です。

---

### I. アーキテクチャ説明書の基本的な構成と分量

アーキテクチャ説明書は、システムのモデルの**ビュー（View）**として表されます。ビューは、対応するモデル（ユースケースモデル、分析モデルなど）の「抜粋」または「一部」であり、**アーキテクチャ上重要な要素**に限定して記述されます。

*   **構成要素**: ユースケース、サブシステム、インターフェイス、一部のクラスとコンポーネント、ノード、コラボレーションなどが含まれます。
*   **分量**: 大規模なシステムの場合、アーキテクチャ説明書の絶対的な文章量に基準はありませんが、経験的には50〜100ページ程度が適切だとされています。すべてを網羅するのが目的ではなく、高次元のビューにすぎません。
*   **アーキテクチャではないもの**: テストケースやテストプロシージャは、アーキテクチャ説明書には含まれません。これらはアーキテクチャの妥当性を検査するため、または検証するために必要な情報ですが、アーキテクチャそのものではないからです。
*   **維持管理**: アーキテクチャ説明書はシステムの生涯にわたって最新に維持され、アーキテクチャに関係する変更や追加が反映されます。

### II. 方向づけフェーズ（Inception Phase）での焦点と内容

方向づけフェーズの主な目標は、「開発企画書（ビジネスケース）の確立」と「システムの実現性の確立」であり、このフェーズで作成されるアーキテクチャ説明書は**「アーキテクチャ候補の説明書」**または**「最初の草案」**という位置づけになります。

この段階では、後のフェーズで安定したアーキテクチャを作成できるという確信を得ることが目標であり、**実装や設計の詳細は最小限**にとどめられます。

方向づけフェーズの序盤から中盤で記載しておくべき内容は、主に**システムの責任範囲**の設定、**致命的リスクの軽減**、および**技術的な実現性**に関わるものです。

| 成果物/ビュー | 求められる内容（方向づけフェーズでの焦点） | 根拠となる資料 |
| :--- | :--- | :--- |
| **アーキテクチャ候補の概要** | **システムの新規部分、リスクがある部分、あるいは難しい部分**について記述されます。アーキテクチャが大丈夫だという「根拠」に基づいて、その妥当性を判定できるレベルが求められます。 | |
| **ユースケースモデルのビュー** | **最も重要なユースケース**（システムの中心的な機能を形成するユースケース）の記述の一部。**致命的リスクの理解に関わるユースケース**が選ばれます。通常、ユースケース群全体のわずか10%未満が詳細化され、その記述全体がビューに入る場合があります。 | |
| **補足仕様書の内容** | **アーキテクチャにとって重要な機能外要求**。これには、プラットフォーム、システムソフトウェア、ミドルウェアの選択といったアーキテクチャに影響を及ぼす一般的な制約が含まれます。 | |
| **設計モデル/配置モデルのビュー** | **略述された最初の輪郭**。設計の輪郭にすぎないとしても、サブシステムとクラス間のインターフェイスの洗い出し（および操作のいくつかの定義）など、**アーキテクチャの中心部分となる要素**が重要です。提案したシステムが分散される場合は、**性能を要求されるノードやノード間の接続に限定した小規模の配置モデル**が含まれることがあります。 | |
| **汎用の設計メカニズム** | アーキテクチャの土台となる技術の選択が適切か、効率的か、信頼性が高いかなどの**実現性を確信するための根拠**を記述します。ミドルウェアやシステムウェアのフレームワークの選択が含まれます。 | |

このフェーズの目標は、プロジェクトが実現可能であること、そしてアーキテクチャの作成が可能なことを確信できるようにすることで、この「アーキテクチャ候補の説明書」は、次の推敲フェーズの作業を導くための基礎となります。

### III. その他のフェーズでの役割と内容の概略

アーキテクチャ説明書は、推敲フェーズで完成し、それ以降のフェーズでは主に維持・更新されます。

#### 1. 推敲フェーズ（Elaboration Phase）

推敲フェーズの最大の目標は、**実行可能なアーキテクチュラルベースライン（Architectural Baseline）**を確立することです。このベースラインがシステムの「骨格」となり、アーキテクチャ説明書は、このベースラインのモデルのビューを含みます。

| 焦点となる作業 | アーキテクチャ説明書の内容の進展 | 根拠となる資料 |
| :--- | :--- | :--- |
| **アーキテクチャの安定化** | アーキテクチャ説明書は、このフェーズの終わりまでに**安定**します。システムの生涯を通じて開発チーム全体に指針を与える**「標準」**となります。 | |
| **モデルのビューの充実** | **すべてのモデル**（ユースケース、分析、設計、実装、配置）のビューが含まれ、**アーキテクチャ上重要な要素が安定した状態**で記述されます。 | |
| **汎用メカニズムの明確化** | 永続性、セキュリティ特性、トランザクション管理などの**共通する要求**を取り扱うための汎用の設計メカニズム、および使用する**アーキテクチャパターン**の説明を記述します。 | |
| **実装・配置の具体化** | 配置モデルのビューには、**コンポーネントのノードへのマッピング**など、実装に必要なあらゆる観点が重要性が高いため表されます。 | |

推敲フェーズが終わる時点（ライフサイクルアーキテクチャのマイルストーン）で、アーキテクトは「これで、技術的には重大な問題に直面することなく、システムを作成できる」ということを保証する説明書を提出します。

#### 2. 作成フェーズ（Construction Phase）

作成フェーズでは、アーキテクチャの骨格に「肉づけ」を行い、残りの機能の実装に焦点が移ります。

*   **更新の必要性**: アーキテクチャは安定しているため、アーキテクチャ説明書自体は**あまり増加せず**、**関連する部分の更新**が行われるだけです。
*   **アーキテクトの役割**: アーキテクトは、作成がアーキテクチャに基づいて**忠実に行われるように注意を払い**、必要ならば、作成作業で発生する変更を取り入れるためにアーキテクチャを変更します。変更が必要な場合は、新しい抽象クラスやインターフェイスの検出、再利用可能なコンポーネントのアップグレード、プロセス構造の割り当て直しといった**副次的な変更**が反映されます。

#### 3. 移行フェーズ（Transition Phase）

移行フェーズでは、システムをユーザー環境で正しく動作させることに焦点を当て、欠陥の修正などが中心となります。

*   **維持と保証**: アーキテクチャ説明書は、このフェーズでも維持され、アーキテクトは、問題と欠陥の解決が必ずアーキテクチャに基づいて行われるように「追跡モード」で作業します。アーキテクチャの整合性の保証と維持が主な目的であり、ときには小さなアーキテクチャ上の変更を検討することもあります。

---

アーキテクチャ説明書は、ソフトウェアシステムを建物に例えるなら、設計の詳細（詳しい構造図面）ではなく、**建物の全体的な構図（縮尺モデル）**を提示する役割を果たします。これにより、開発の初期段階でシステムの最も重要な骨格（アーキテクチャ）を確立し、将来の変更に対する弾力性（許容度）を持たせることを可能にします。
